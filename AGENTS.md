# AGENTS.md

## 1. 文档目的

本文档用于约定 MyAITodo 仓库在后续维护、协作和继续开发中的统一执行规则，确保项目在个人开发场景下保持稳定、清晰、可持续演进。

## 2. 项目定位

- 项目名称：MyAITodo
- 项目类型：个人 AI Todo 全栈网页项目
- 当前阶段：V1 已完工，可继续做增量迭代
- 协作原则：轻量、清晰、可验证、可提交

## 3. 开发原则

- 优先保证主链路可运行，再扩展新能力
- 优先保证真实联调通过，不以单纯代码检查代替运行验证
- 优先小步提交，避免大范围无边界改动
- 优先修复用户可见问题，如运行错误、乱码、异常交互、接口失败

## 4. 测试标准

以后对“完成”或“测试通过”的定义，统一分为三层：

### 4.1 代码检查通过

- 前端 `lint` 通过
- 前端 `build` 通过
- 后端基础编译或导入检查通过

### 4.2 接口测试通过

- 后端接口可通过真实 HTTP 调用验证
- 核心接口返回码与返回结构正确
- 数据能真实写入数据库

### 4.3 真实联调通过

- 前端页面真实可访问
- 前端请求真实打到后端
- 后端真实写入数据库
- 用户实际操作链路可完成

只有三层都通过，才可以对外表述为“本轮完成”。

## 5. Git 规范

### 5.1 分支策略

- 个人开发场景下，以主分支为主
- 如需临时分支，默认使用 `codex/` 前缀

### 5.2 提交原则

- 一个提交只解决一类问题或一个阶段性目标
- 提交前先检查 `git status`
- 提交说明必须简洁明确，使用英文类型前缀

### 5.3 Commit Message 规范

推荐格式：

```text
feat(module): add xxx
fix(module): fix xxx
docs(module): update xxx
refactor(module): improve xxx
test(module): add xxx
chore(module): clean xxx
```

## 6. 文档维护规则

- `README.md` 保持项目当前真实状态
- `docs/` 下文档用于沉淀正式输出
- 若项目状态变化，必须同步更新对应文档

## 7. 改动优先级

当多个任务并存时，处理优先级如下：

1. 运行错误
2. 前后端真实联调失败
3. 数据错误或鉴权错误
4. 用户可见乱码与交互问题
5. 新功能开发
6. 样式微调与非关键优化

## 8. Bad Case 要求

后续每轮核心功能修改后，至少补测以下异常场景中的相关项：

- 未登录访问
- 非法 Token
- 空输入
- 重复创建
- 错误参数
- 不存在资源
- 接口失败后的前端提示

## 9. 当前结论

本仓库当前已进入“V1 完工，允许继续增量迭代”的状态。
后续所有开发工作，默认在不破坏现有主链路的前提下进行。
